Slide 2: O Desafio da "Base Suja"

Muitos temem que o GUID v7 não funcione em bases legadas sem um REBUILD total

O medo comum: "Se eu trocar para v7 agora, a fragmentação das 100K linhas v4 vai contaminar as novas inserções. Preciso rebuildar tudo primeiro, ou o ganho demora a aparecer."

O teste: Simulamos uma base com 100.000 linhas de GUID v4 (índice B-Tree totalmente fragmentado) e começamos a inserir novas linhas usando Guid.CreateVersion7(), sem tocar nas linhas antigas.

Pergunta-chave: Em quantas inserções novas o V7 Hybrid (apenas trocar a linha de código) se torna equivalente ao V7 Full Rebuild (migração completa de todos os IDs)?
Slide 3: O Resultado — Convergência Imediata

V7 Hybrid praticamente empata com Full Rebuild desde os primeiros 10K inserts

Resultado principal: Com apenas 10.000 novos inserts (10% do tamanho da base legada), o V7 Hybrid já performa praticamente igual ao Full Rebuild — diferença de apenas 0,2ms.

Comparação das 3 estratégias ao longo do teste:
  • 10K inserts: V4: 76,1ms | Hybrid: 16,0ms | Full: 15,8ms (Hybrid 4,8× mais rápido que V4)
  • 100K inserts: V4: 989,1ms | Hybrid: 195,0ms | Full: 196,3ms (Hybrid 5,1× mais rápido que V4)
  • 350K inserts: V4: 4.818,6ms | Hybrid: 766,6ms | Full: 712,8ms (Hybrid 6,3× mais rápido que V4)
Diferença Hybrid vs Full: ~20ms em média (imperceptível)
Ganho Hybrid vs V4: 4× a 6× consistente em todo o range

Conclusão: Não existe "aquecimento" necessário. O V7 Hybrid entrega performance equivalente ao Full Rebuild desde o primeiro insert — e ambos destroem o V4.

Slide 4: Por que a "cura" é tão rápida?

Mecânica do B-Tree: isolamento de páginas novas vs antigas

Ordenação temporal: GUID v7 utiliza um timestamp Unix de 48 bits nos primeiros bytes, codificado em big-endian. Isso significa que cada novo v7 gerado é maior que todos os v7 anteriores.

B-Tree append-friendly: No índice B-Tree do banco de dados, inserções ordenadas vão para o final da estrutura — novas páginas, preenchidas sequencialmente. Não há split de páginas intermediárias nem reorganização de blocos existentes.

Isolamento físico: As 100K linhas v4 fragmentadas ficam nas páginas antigas. As novas linhas v7 vão para páginas novas no final. Elas não lutam por espaço.

O que a fragmentação v4 afeta:
  • ❌ NÃO afeta: o caminho de escrita (INSERT) das novas chaves v7
  • ✅ Afeta: buscas (SELECT) por IDs v4 antigos — mais page reads
Por isso o ganho de performance em inserções aparece instantaneamente, mesmo com a base "suja".

Slide 6: Qual a diferença real entre Hybrid e Full Rebuild?

Gap absoluto: ~10ms a 40ms — imperceptível em produção

Comparação direta Hybrid vs Full Rebuild (em milissegundos):
  • 10K inserts: V4: 76,1ms | Hybrid: 16,0ms | Full: 15,8ms → Gap: +0,2ms
  • 100K inserts: V4: 989,1ms | Hybrid: 195,0ms | Full: 196,3ms → Gap: -1,3ms (Hybrid mais rápido!)
  • 350K inserts: V4: 4.818,6ms | Hybrid: 766,6ms | Full: 712,8ms → Gap: +53,8ms
Gap médio (Hybrid - Full): ~20ms
Gap máximo encontrado: ~54ms (em 350K inserts)

Contexto: Em um sistema real com latência de rede (10–100ms), cache de ORM, serialização JSON, etc., essa diferença de 10–50ms entre Hybrid e Full Rebuild é estatisticamente irrelevante.

Insight: O V7 Hybrid entrega 98% da performance do Full Rebuild com 0% do risco e do trabalho de migração. E ambos destroem o V4 em qualquer escala.

Slide 7: Conclusão Prática

Trocar para Guid.CreateVersion7() entrega o benefício desde o primeiro dia

Mudança necessária: 
// Antes
var id = Guid.NewGuid();

// Depois
var id = Guid.CreateVersion7();
ROI imediato:
  • ✅ 0 schema change — formato 128 bits mantido
  • ✅ 0 migração de dados — IDs antigos coexistem nativamente
  • ✅ 0 downtime — deploy padrão
  • ✅ Ganho de 4× a 6× na performance de INSERT desde a primeira execução
REORGANIZE/VACUUM é opcional: elimina a fragmentação herdada e deixa o índice no estado ótimo, mas os dados mostram que isso é uma otimização incremental, não um pré-requisito.

Risco de rollback: Baixíssimo. Se precisar voltar para v4, basta reverter a linha de código. Os IDs v7 já gerados continuam válidos como GUIDs normais.

Ganho de CPU/IO: Instantâneo. Menos page splits = menos I/O de disco = menor latência de escrita = maior throughput.
